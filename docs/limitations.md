# Limitations — LVCA

Честный обзор того, что LVCA **не умеет** или делает **неидеально** на текущем этапе.

---

## Голос и распознавание речи

### Нет wake-word ("Jarvis, ...")
LVCA не слушает постоянно в фоне. Работает только push-to-talk (нажать кнопку → говорить → отпустить). Непрерывное распознавание с wake-word (типа "Эй, LVCA") не реализовано.

**Обходной путь:** Держать Desktop App открытым, использовать горячую клавишу для push-to-talk.

---

### Качество TTS — "роботизированный голос"
Piper VITS (основной TTS) — это **CPU-модель 30 MB**. Качество речи приемлемо, но заметно отличается от человеческой речи. Русский голос "Irina" средней натуральности.

**Если нужен более живой голос:**
```env
TTS_ENGINE=xtts
TTS_DEVICE=cuda
```
XTTS-v2 значительно лучше, но требует ~2 GB VRAM дополнительно и увеличивает латентность TTS до ~1–2 с.

---

### Клонирование голоса — экспериментально
XTTS-v2 поддерживает zero-shot клонирование голоса. Однако:
- Качество клонирования на русском хуже, чем на английском
- Нужно 20+ минут записи для приемлемого качества
- Иногда звук "плавает" или появляются артефакты

---

### Задержка полного цикла
Типичное время от конца речи до начала голосового ответа: **3–7 секунд**.

Разбивка:
- STT: ~0.5–1.5 с
- LLM без инструментов: ~2–5 с
- LLM с инструментами: ~5–20 с (до 8 шагов)
- TTS: ~0.1–0.5 с

LVCA **не подходит** для разговоров в реальном времени с очень низкой задержкой (< 1 с). Это не голосовой звонок — это асинхронный ассистент.

---

## LLM и понимание

### Ограничения Qwen 2.5 7B
Модель 7B (Q4_K_M квантизация) иногда:
- Неправильно интерпретирует сложные многоступенчатые задачи
- "Галлюцинирует" — выдаёт правдоподобно звучащую, но неверную информацию
- Путается при длинных цепочках инструментов (> 5 шагов)
- Делает ошибки в сложной математике без `code_gen`

**Обходной путь:** Формулировать запросы чётко и конкретно. Использовать `code_gen` для математики и вычислений.

---

### Нет интернета (принципиально)
LVCA работает полностью оффлайн. Инструмент `browser` использует Playwright для просмотра страниц, но:
- Сам LLM не имеет актуальных данных после даты обучения
- `browser` может скрапить страницы, но не может обходить авторизацию, CAPTCHA, динамический JS
- Для актуальных новостей и данных — только через `browser` tool

---

### Контекстное окно и длинные разговоры
Память — скользящее окно (последние ~20 сообщений по умолчанию). При очень длинных диалогах:
- Ранние части разговора забываются
- RAG-контекст не компенсирует потерю conversational context

**Обходной путь:** Увеличить `MEMORY_MAX_MESSAGES=50` (потребляет больше VRAM при каждом запросе).

---

## Desktop Agent и автоматизация

### Работает только на Windows
Desktop Agent использует pyautogui, pygetwindow, pycaw — библиотеки, специфичные для Windows. На macOS и Linux:
- Базовый функционал (клавиатура, мышь, скриншоты) может работать через pyautogui
- Управление звуком (pycaw) — только Windows
- Управление окнами (pygetwindow) — ограниченная поддержка Linux/Mac

---

### Нет видения экрана в реальном времени
Инструмент `vision` делает один статичный скриншот и анализирует его. LVCA **не видит** экран непрерывно и не реагирует на изменения без явного запроса.

---

### Browser automation ограничена
Playwright (headless Chromium) работает хорошо для:
- Поиска в DuckDuckGo
- Открытия URL
- Извлечения текста со страниц

Но **не работает** для:
- Сайтов с CAPTCHA
- Авторизации (без явного ввода паролей)
- Сложных SPA с тяжёлым JS
- Видеопотоков

---

### pyautogui — экранные координаты
Инструменты `click`, `scroll`, `type_text` работают на уровне экрана (абсолютные координаты). Это значит:
- Если разрешение экрана изменилось — координаты могут быть неверными
- Нет интеллектуального поиска UI-элементов (по тексту, aria-label и т.д.)
- Несколько мониторов: нужно явно указывать координаты

---

## Производительность

### Потребление памяти
При одновременной работе всех моделей:
- VRAM: ~7–9 GB (пик при первом LLM-запросе)
- RAM: ~8–12 GB (Docker + Python процессы)
- Рекомендуется 24 GB RAM для комфортной работы

На системах с < 16 GB RAM возможна активация swap, что сильно замедлит работу.

---

### Первый запрос медленный
При первом запросе после запуска:
- Ollama загружает модель в VRAM: ~10–30 с
- STT загружает Whisper: ~5–15 с
- После прогрева — нормальная скорость

---

### Параллельные запросы
Каждый сервис обрабатывает запросы последовательно (один за раз). Если отправить несколько запросов одновременно — они встанут в очередь. Нет горизонтального масштабирования "из коробки" (есть через Kubernetes).

---

## Мониторинг и логирование

### Нет алертов в реальном времени
Prometheus + Grafana собирают метрики, но нет push-уведомлений при деградации (нет интеграции с Slack, Email и т.д.).

---

## Безопасность

### Нет аутентификации на API
Все HTTP endpoints открыты без авторизации. Предполагается, что LVCA работает на localhost или в изолированной сети. Не размещать в публичный интернет без добавления auth-слоя.

---

### Desktop Agent — localhost only, но всё же нативный доступ
Desktop Agent имеет полный доступ к рабочему столу. Несмотря на whitelist и классификацию действий, это нативный процесс с реальными правами. Не запускать на критических серверах.

---

## Чего нет в текущей версии

| Функция | Статус | Примечание |
|---------|--------|-----------|
| Wake-word | Нет | Только push-to-talk |
| Многопользовательский режим | Нет | Один локальный пользователь |
| История разговоров в файл | Нет | Только в памяти (Redis) |
| GUI для управления RAG | Нет | Только CLI / API |
| Автообновление моделей | Нет | Ручное через make pull-models |
| Голосовые профили (несколько голосов) | Частично | Только через конфиг |
| Поддержка macOS / Linux Desktop Agent | Частично | pyautogui работает, остальное нет |
| Интеграция с календарём / почтой | Нет | Только через browser tool |
| Мобильное приложение | Нет | Только Desktop App (Windows) |
| Streaming STT в реальном времени | Частично | WebSocket есть, VAD не идеален |
| Fine-tuning LLM | Нет | Только инференс |
