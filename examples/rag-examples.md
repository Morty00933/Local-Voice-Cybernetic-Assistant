# RAG Examples — Примеры работы с базой знаний

RAG (Retrieval-Augmented Generation) позволяет LVCA отвечать на вопросы, используя твои документы.

---

## Как работает RAG в LVCA

```
1. Пользователь добавляет документы в knowledge/
2. make index-docs → документы разбиваются на чанки (800 токенов)
3. Каждый чанк vectorизуется через nomic-embed-text (768-dim)
4. Векторы сохраняются в Qdrant

При запросе:
5. Запрос пользователя → вектор
6. Cosine search в Qdrant → top-6 релевантных чанков (score > 0.5)
7. Найденный контекст инжектируется в промпт LLM
8. Qwen отвечает используя контекст документов
```

---

## Быстрый старт

```bash
# 1. Добавить документы
cp my_manual.md knowledge/
cp project_docs.txt knowledge/
cp technical_spec.pdf knowledge/

# 2. Запустить индексацию
make index-docs

# 3. Задать вопрос (Brain автоматически ищет контекст)
curl -X POST http://localhost:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"text": "Что написано в руководстве?", "session_id": "s1"}'
```

## Сценарий 1: Техническая документация проекта

**Документ (`knowledge/api_docs.md`):**
```markdown
# API Endpoints

## POST /api/users
Создаёт нового пользователя.
Параметры: name (string), email (string), role (admin|user|guest)
Возвращает: {"id": "uuid", "status": "created"}

## GET /api/users/{id}
Получает пользователя по ID.
Возвращает: {"id": "uuid", "name": "...", "email": "...", "role": "..."}

## DELETE /api/users/{id}
Удаляет пользователя. Требует роль admin.
```

**После `make index-docs`:**

```
Вопрос: "Как создать пользователя через API?"
→ Для создания пользователя используйте POST /api/users.
  Передайте JSON с полями name, email и role (admin, user или guest).
  В ответ получите id нового пользователя и статус created.

Вопрос: "Какие роли поддерживает система?"
→ Система поддерживает три роли: admin, user и guest.
  Роль admin требуется для удаления пользователей (DELETE /api/users/{id}).

Вопрос: "Что вернёт DELETE /api/users/123?"
→ Эндпоинт DELETE /api/users/{id} удаляет пользователя. Требует роль admin.
  В документации не указан формат ответа — рекомендую проверить в Swagger UI.
```

```markdown
# Процедура инцидент-менеджмента
```

```
Вопрос: "Что делать при критическом инциденте?"
```

## Сценарий 5: Индексация через API

Можно добавлять документы программно без перезапуска:

```bash
# Добавить один документ через API
curl -X POST http://localhost:8002/api/index \
  -H "Content-Type: application/json" \
  -d '{
    "text": "# Наш продукт\n\nОписание продукта...",
    "filename": "product.md"
  }'

# Ответ:
# {"status": "indexed", "chunks": 3, "filename": "product.md"}
```

```python
# Python пример
import httpx

async def index_document(text: str, filename: str):
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "http://localhost:8002/api/index",
            json={"text": text, "filename": filename}
        )
        return response.json()

# Использование:
await index_document(open("my_docs.md").read(), "my_docs.md")
```

## Как RAG влияет на ответы

**Без RAG** (нет релевантных документов):
```
Вопрос: "Какой дедлайн для задачи PROJ-456?"
→ Я не имею информации о конкретной задаче PROJ-456.
  Можете проверить в Jira или предоставить мне детали задачи.
```

**С RAG** (документы проиндексированы):
```
Вопрос: "Какой дедлайн для задачи PROJ-456?"
→ [RAG нашёл в your_tasks.md: "PROJ-456: дедлайн 15 марта 2025"]
  Задача PROJ-456 имеет дедлайн 15 марта 2025.
```

## Настройка точности RAG

Если RAG находит нерелевантный контекст или не находит нужный:

```env
# В .env

# Снизить порог — больше результатов (менее точно)
RAG_SCORE_THRESHOLD=0.3   # default: 0.5

# Увеличить количество чанков
RAG_TOP_K=10              # default: 6

# Уменьшить размер чанков (лучше для коротких фактов)
CHUNK_SIZE=400            # default: 800

# Увеличить overlap (лучше для связных текстов)
CHUNK_OVERLAP=200         # default: 120
```

После изменения настроек — переиндексировать:
```bash
curl -X DELETE http://localhost:6333/collections/lvca_knowledge
make index-docs
```

## Советы по подготовке документов

**Форматы которые работают лучше:**
- `.md` — лучший выбор: Markdown-aware chunking сохраняет структуру
- `.txt` — хорошо для простого текста
- `.pdf` — работает, но качество зависит от PDF (сканы плохо читаются)

**Структурируй документы для лучшего RAG:**
```markdown
# Раздел 1 (заголовок — граница чанка)
Содержимое раздела...

## Подраздел 1.1
Конкретная информация здесь...
```

**Избегай:**
- Огромных таблиц без текстового описания
- Скан-PDF (без OCR)
- Дублирующегося контента
- Информации которая часто меняется (лучше обновлять файл и переиндексировать)

| Формат | Поддержка | Качество |
|--------|-----------|---------|
| `.md` | Да | Отличное (Markdown-aware chunking) |
| `.txt` | Да | Хорошее |
| `.pdf` (текстовый) | Да | Хорошее |
| `.pdf` (скан) | Нет | — (нет OCR) |
| `.docx` | Нет | — |
| `.html` | Нет | — |
